## 결정 1: 네이버 Maps JavaScript API v3 사용

**상태:** 확정

**맥락**
사용자가 네이버지도 기반 서비스를 명시적으로 요청했다.

**결정**
네이버 Maps JavaScript API v3 (웹 Dynamic Map)을 사용한다. index.html에 스크립트 태그로 로드하되, Client ID는 환경변수(VITE_NAVER_CLIENT_ID)에서 읽는다.

**고려한 대안**
- 카카오맵 API — 사용자 요구사항과 불일치
- Google Maps API — 사용자 요구사항과 불일치
- Leaflet + 타일 서버 — 네이버지도 타일을 직접 쓸 수 없음

**결과**
사용자가 이미 API 키를 보유하고 있으므로 .env.example에 placeholder를 제공하고 README에 설정 안내를 포함한다.

## 결정 2: localStorage 기반 데이터 저장

**상태:** 확정

**맥락**
사용자가 백엔드 없이 로컬스토리지 기반 데이터 저장을 명시했다.

**결정**
브라우저 localStorage에 JSON 직렬화하여 장애물 데이터를 저장한다. key는 `obstacles`로 통일한다.

**고려한 대안**
- IndexedDB — 대용량에 유리하나 이 규모에서는 과잉. 향후 마이그레이션 가능
- sessionStorage — 탭 종료 시 데이터 손실
- Cookie — 용량 제한(4KB), 비적합

**결과**
약 5MB 용량 한계가 있으므로 이미지 첨부 등은 제외한다. 데이터 구조를 효율적으로 설계하면 수천 건의 장애물 데이터를 충분히 저장 가능하다. 백업/내보내기 기능은 이번 스코프에서 제외한다.

## 결정 3: React + Vite 기술 스택

**상태:** 확정

**맥락**
프론트엔드 기술 스택을 결정해야 한다. 사용자가 React + Vite를 선택했다.

**결정**
React + Vite로 구현한다. JSX 기반 컴포넌트 구조로 개발하며, 추가 CSS 프레임워크 없이 순수 CSS로 스타일링한다.

**고려한 대안**
- 바닐라 HTML/CSS/JS — 빌드 도구 불필요하나 컴포넌트 재사용성 낮음
- Vue + Vite — 사용자가 React를 선택
- Next.js — SSR이 불필요한 SPA이므로 과잉

**결과**
Vite의 빠른 HMR과 React의 컴포넌트 구조를 활용한다. 네이버지도 스크립트는 index.html에 직접 로드하거나 동적 로드 방식을 사용한다.

## 결정 4: 카테고리 + 위험도 이중 분류 체계

**상태:** 확정

**맥락**
장애물 분류 방식을 결정해야 한다. 사용자가 카테고리와 위험도를 함께 사용하겠다고 결정했다.

**결정**
각 장애물에 카테고리(종류)와 위험도(심각성)를 모두 입력한다.
- 카테고리: 기본 6종(공사중, 도로파손, 침수, 낙석, 보행위험, 기타) + 사용자 자유 추가 가능
- 위험도: 3단계(위험, 주의, 안전)
- 마커 색상은 위험도에 따라 결정: 위험(#FF4444), 주의(#FFA500), 안전(#44BB44)

**고려한 대안**
- 카테고리만 사용 — 심각성 구분 불가
- 위험도만 사용 — 종류 구분 불가
- 5단계 위험도 — 3단계로 충분히 직관적

**결과**
필터링 시 카테고리와 위험도를 조합하여 사용할 수 있다. 마커 시각적 구분이 명확해진다.

## 결정 5: 반응형 전략

**상태:** 확정

**맥락**
반응형 UI가 요구사항이다. 모바일과 데스크탑 모두 지원해야 한다.

**결정**
모바일 퍼스트 CSS 미디어쿼리 기반 반응형 레이아웃을 적용한다.
- 모바일(~767px): 지도 전체 화면 + 하단 시트형 목록 패널
- 데스크탑(1024px~): 지도 + 고정 좌측 사이드바

**고려한 대안**
- 별도 모바일 페이지 — 유지보수 부담 증가
- Tailwind CSS — 추가 의존성 없이 순수 CSS로 충분히 구현 가능

**결과**
React 컴포넌트 내에서 CSS 미디어쿼리 또는 CSS 모듈을 사용하여 반응형을 구현한다.
