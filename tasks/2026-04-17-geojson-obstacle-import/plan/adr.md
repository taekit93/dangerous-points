# ADR — GeoJSON 장애물 임포트

**작성일:** 2026-04-17
**작성자:** planner (초안) / reviewer (검토 후 보완)

## 결정 1: GeoJSON 파싱을 별도 유틸리티 함수로 분리

**상태:** 확정

**맥락**
GeoJSON 파싱은 UI 로직과 독립적이며, 단위 테스트가 필요하다. 컴포넌트 내부에 파싱 로직을 넣으면 테스트와 재사용이 어렵다.

**결정**
`src/utils/geojson.js`에 파싱/검증/변환 함수를 분리한다.

**고려한 대안**
- 컴포넌트 내부에 직접 구현 — 테스트 어려움, 재사용 불가
- 외부 라이브러리(@turf/turf 등) 도입 — 현재 프로젝트가 의존성 최소화 전략이므로 과도함

**결과**
파싱 로직이 독립적으로 테스트 가능하고, 향후 다른 포맷 지원 시 확장 용이.

## 결정 2: 임포트 UI를 모달 형태로 구현

**상태:** 확정

**맥락**
임포트 기능은 일시적 작업이다. 기존 ObstacleForm과 같은 overlay/panel 패턴이 이 프로젝트의 UI 패턴이다.

**결정**
ObstacleForm과 동일한 overlay + panel 패턴의 모달 컴포넌트(GeoJsonImportModal)로 구현한다.

**고려한 대안**
- 사이드바에 통합 — 공간이 부족하고 기존 FilterBar/MarkerList와 충돌
- 별도 페이지 — 현재 SPA 단일 페이지 구조에 맞지 않음

**결과**
기존 UI 패턴과 일관성 유지. 사용 완료 후 닫으면 원래 화면으로 복귀.

## 결정 3: 임포트 버튼을 Header 우측에 배치

**상태:** 확정 (designer 단계에서 변경)

**맥락**
임포트는 장애물 등록의 한 방법이다. 초안에서는 DrawToolbar에 배치를 결정했으나, designer가 DrawToolbar는 드로잉 모드 전용이므로 데이터 임포트와 책임이 다르다고 판단하여 Header 우측으로 변경했다.

**결정**
Header 우측 끝(count 배지 오른쪽)에 "GeoJSON 가져오기" 버튼을 배치한다. 모바일(480px 이하)에서는 아이콘만 표시한다.

**고려한 대안**
- DrawToolbar에 배치 — 드로잉 모드 전용 영역이므로 데이터 임포트와 책임 불일치
- 사이드바 상단에 배치 — 모바일에서 사이드바가 숨겨져 있어 접근성 떨어짐

**결과**
Header에서 항상 접근 가능하며, DrawToolbar의 드로잉 전용 책임을 유지.

## 결정 4: GeoJSON properties 자동 매핑 (빈칸 폴백)

**상태:** 확정

**맥락**
외부 GeoJSON 데이터의 properties 키 이름은 표준이 없다. title, name, 제목 등 다양한 키가 사용될 수 있다.

**결정**
우선순위 기반 자동 매핑을 적용한다. 매핑 실패 시 빈칸으로 두고, 공통 메타데이터 폼에서 사용자가 채울 수 있다:
- title: properties.title -> properties.name -> properties.제목 -> ""
- category: properties.category -> properties.카테고리 -> ""
- dangerLevel: properties.dangerLevel -> properties.danger -> properties.위험도 -> ""
- description: properties.description -> properties.설명 -> ""

**고려한 대안**
- 사용자가 매핑 규칙을 직접 설정 — 초기 구현에서는 과도한 복잡성
- 모든 properties를 description에 덤프 — 정보 활용도 낮음

**결과**
대부분의 일반적인 GeoJSON 데이터에서 자동으로 속성이 매핑됨. 매핑되지 않는 경우 공통 메타데이터 폼으로 보완.

## 결정 5: FeatureCollection 일괄 등록 + 공통 메타데이터 폼

**상태:** 확정

**맥락**
FeatureCollection에 여러 Feature가 포함될 때, 각 Feature를 개별 확인/등록할지 한 번에 전체 등록할지 결정이 필요했다.

**결정**
FeatureCollection의 모든 Feature를 한 번에 전체 등록한다. 공통 메타데이터 폼을 제공하여 properties 자동 매핑이 실패한 필드에 일괄 적용할 값을 입력받는다.

**고려한 대안**
- 개별 Feature 확인/수정 후 선택적 등록 — 대량 데이터에서 UX가 번거로움
- 자동 매핑만 제공하고 수정 불가 — 속성이 없는 GeoJSON에서 불편

**결과**
대량 임포트의 효율성을 유지하면서, 공통 메타데이터 폼으로 최소한의 속성 보완이 가능.

## 결정 6: 중복 검사 없이 그대로 등록

**상태:** 확정

**맥락**
동일 좌표에 이미 장애물이 있는 경우 중복으로 간주하여 건너뛸지 결정이 필요했다.

**결정**
중복 검사를 수행하지 않고 그대로 등록한다. 동일 좌표의 장애물이 있더라도 별도 항목으로 저장한다.

**고려한 대안**
- 좌표 기반 중복 검사 — 좌표 비교 로직(부동소수점 오차 처리 등)이 복잡하고, "같은 위치의 다른 장애물"이 정당한 경우도 있음
- 사용자에게 중복 경고 후 선택 — 구현 복잡성 대비 효용이 낮음

**결과**
구현이 단순하며, 사용자가 필요 시 수동으로 중복을 정리할 수 있음.
