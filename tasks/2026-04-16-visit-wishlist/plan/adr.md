# ADR — 방문 위시리스트

**작성일:** 2026-04-16
**작성자:** planner (초안) / reviewer (검토 후 보완)

## 결정 1: 위시리스트 데이터를 장애물과 별도 localStorage 키로 저장

**상태:** 확정

**맥락**
기존 앱은 `obstacles` 키로 localStorage에 장애물 데이터를 저장한다.
위시리스트 데이터를 같은 키에 type 필드로 구분할지, 별도 키로 분리할지 결정해야 한다.

**결정**
`wishlist`라는 별도 localStorage 키를 사용한다.
storage.js 패턴을 복제하여 wishlistStorage.js를 만든다.

**고려한 대안**
- 대안 A: 같은 `obstacles` 키에 `itemType: 'wishlist'` 필드 추가 — 기존 데이터 구조 오염, 필터링 복잡도 증가, 기존 기능에 영향 가능성
- 대안 B: 범용 storage 모듈에 키를 파라미터로 전달 — 좋은 접근이나 기존 storage.js 변경 필요, 최소 diff 원칙 위반

**결과**
데이터가 완전히 분리되어 기존 장애물 기능에 영향 없음. 코드 중복이 소량 발생하나 독립성이 더 중요.

## 결정 2: 모드 전환 방식으로 탭 UI 채택

**상태:** 확정

**맥락**
장애물 등록과 위시리스트 등록을 동시에 활성화하면 지도 클릭 이벤트 충돌이 발생한다.
사용자가 현재 어떤 작업 중인지 명확히 알 수 있어야 한다.

**결정**
Header 영역에 "장애물" / "위시리스트" 탭을 추가하여 현재 활성 모드를 전환한다.
각 모드에서 지도 클릭 시 해당 타입의 등록 폼이 열린다.
두 타입의 마커는 항상 지도에 동시 표시되며, 모드는 "등록" 동작에만 영향을 준다.

**고려한 대안**
- 대안 A: 플로팅 버튼으로 모드 전환 — 발견성이 낮고 현재 모드를 시각적으로 파악하기 어려움
- 대안 B: 사이드바에서만 관리, 지도 클릭 등록 없음 — 지도 앱의 직관적 UX를 해침

**결과**
직관적인 모드 표시, 지도 클릭 충돌 방지, 기존 장애물 UX 변경 최소화.

## 결정 3: 위시리스트 마커는 별 모양 아이콘 + 파란색/금색 사용

**상태:** 확정

**맥락**
기존 장애물 마커는 위험도별 빨강/주황/초록 핀 모양이다.
위시리스트 마커가 장애물과 시각적으로 확실히 구분되어야 한다.

**결정**
위시리스트 마커는 별(star) 모양 SVG 아이콘을 사용한다.
- 미방문 (visitCount === 0): 파란색 별
- 방문완료 (visitCount >= 1): 금색 별

**고려한 대안**
- 대안 A: 같은 핀 모양에 다른 색상 — 장애물 마커와 혼동 가능성
- 대안 B: 하트 모양 — 위시리스트 맥락에서 적절하나, 별이 더 범용적

**결과**
장애물(핀)과 위시리스트(별)가 모양 자체로 즉시 구분 가능. 방문 상태도 색상으로 직관 파악.

## 결정 4: 후기는 다중 작성 가능, 방문 상태와 무관하게 작성 허용

**상태:** 확정 (2차 수정)

**맥락**
후기를 단일/다중으로 할지, 방문 완료 후에만 허용할지 결정해야 한다.

**결정**
위시리스트 항목에 `reviews` 배열을 포함한다. 여러 개의 후기를 작성할 수 있다.
방문 상태와 무관하게 언제든 후기를 작성할 수 있다 (PRD 원래 명세로 복귀).
각 후기에는 선택적으로 `visitNum` (몇 번째 방문 후기인지, 1~9 고정 범위)를 기록할 수 있다.

**[reviewer 보완]** 필드명이 PRD의 `content`/`visitNumber`에서 구현 시 `text`/`visitNum`으로 변경됨. PRD 업데이트 필요.

**변경 이력**
- 초기: PRD 명세대로 방문 무관 작성 허용
- 1차 수정: 디자인 단계에서 visitCount > 0 조건 추가 (open-questions.md 참조)
- 2차 수정: reviewer 지적에 따라 PRD 명세와 불일치 해소, visitCount 조건 제거 (방문 무관 작성으로 복귀)

**고려한 대안**
- 대안 A: 단일 review + rating 필드 — 구현이 단순하나 사용자가 여러 번 방문 시 기록 관리 불가
- 대안 B (기각): visitCount > 0 일 때만 후기 폼 활성화 — PRD 명세와 불일치, 미방문 상태에서 메모를 남기고 싶은 경우를 지원 못함
- 대안 C (채택): 방문 무관 작성 허용 — PRD 명세 준수, 미방문 메모 지원

**결과**
다중 후기 시스템. 방문 상태와 무관하게 자유롭게 후기 작성 가능. 여러 차례 방문 후기를 순서대로 관리 가능.

## 결정 5: 위시리스트 전용 카테고리 사용

**상태:** 확정

**맥락**
기존 장애물에는 type 필드(point, line, polygon)가 있다.
위시리스트 카테고리를 장애물 카테고리와 공유할지, 별도로 정의할지 결정해야 한다.

**결정**
위시리스트 전용 카테고리를 사용한다: 맛집, 카페, 관광지, 공원, 쇼핑, 문화/예술, 기타.
카테고리 목록은 상수 배열로 정의하여 폼과 필터에서 공유한다.

**고려한 대안**
- 대안 A: 장애물과 동일 카테고리 공유 — 장애물 타입(point/line/polygon)은 방문 장소 맥락에 맞지 않음
- 대안 B: 사용자 정의 카테고리 (자유 입력) — 일관성 부족, 필터링 복잡도 증가

**결과**
방문 장소에 적합한 직관적 카테고리. 필터링이 깔끔하고 사용자 경험이 일관적.
